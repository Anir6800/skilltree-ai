# AI Pipeline Storage & Execution Architecture

SkillTree AI relies heavily on asynchronous processing and AI integration to evaluate user code, provide feedback, and detect LLM-generated submissions. This document outlines the storage mechanisms and data flow that support this pipeline.

## 1. Pipeline Overview

The AI evaluation pipeline handles computationally expensive tasks (like running sandboxed code) and network-bound tasks (like querying an LLM for code review). It ensures the main Django request/response cycle remains extremely fast.

The core tools involved are:
- **Django**: API entry points and state management.
- **Celery & Redis**: Task queuing, distributed execution, and result caching.
- **LM Studio / AI Models**: The inference engine processing the prompts.

## 2. Core Storage Models

The pipeline relies on several highly normalized tables to store the outcomes of an evaluation.

### A. ExecutionTask
Tracks the status of asynchronous jobs.
- **`celery_task_id`**: UUID bridging Django with Celery.
- **`status`**: State machine (`PENDING`, `EXECUTING`, `COMPLETED`, `FAILED`).
- **`result_log`**: Raw stdout/stderr from the sandbox.

### B. EvaluationResult
Stores the qualitative and quantitative feedback from the AI.
- **`score`**: Integer (0-100) determining XP payouts.
- **`passed`**: Boolean flag indicating quest success.
- **`feedback`**: JSON blob containing categorized feedback (e.g., `strengths`, `areas_for_improvement`).
- **`test_results`**: JSON array mapping to specific unit tests run against the code.

### C. StyleReport
Stores static analysis and AI-driven style suggestions.
- **`readability_score`**: Float.
- **`style_issues`**: JSON array of line-specific linting or style critiques.
- **`suggestions`**: Actionable refactoring advice.

### D. DetectionLog
Tracks the probability that a submission was generated by an AI rather than a human.
- **`ai_probability`**: Float (0.0 to 1.0).
- **`flag_reason`**: String denoting why it was flagged (e.g., "Abnormal typing speed + High AI similarity").
- **`trigger_phrases`**: JSON array of suspicious code patterns.

## 3. Data Flow and State Management

When a user submits code via a `QuestSubmission`:

1. **Initial Save**: The submission is saved to PostgreSQL with `status="PENDING"`.
2. **Task Dispatch**: Django calls `evaluate_submission.delay(submission_id)` which pushes a message to Redis.
3. **Execution Phase**: A Celery worker picks up the task, updates the state to `EXECUTING`, and runs the code against unit tests.
4. **AI Phase**: If tests pass, the worker constructs an AI prompt (including the code, quest context, and user history).
5. **AI Interaction**: The worker queries the AI (e.g., via LM Studio). To prevent locked workers, strict timeouts are enforced, though high time limits (e.g., 30 mins) are allowed for slower local LLMs.
6. **Data Storage**: The AI returns a structured JSON payload. The worker parses this and creates the `EvaluationResult`, `StyleReport`, and `DetectionLog` records.
7. **Completion**: The `QuestSubmission` is marked `COMPLETED` (or `FAILED` if tests/AI failed), and XP is distributed via PostgreSQL signals.

## 4. Resilience and Error Handling

To maintain a robust pipeline, several safeguards are implemented at the database level:

- **Idempotency**: Celery tasks are designed to be idempotent. If a worker crashes midway, the task can be retried without duplicating XP rewards, as the system checks the `QuestSubmission` status before applying side effects.
- **JSON Validation**: All AI responses are validated against expected Pydantic/JSON schemas before being saved into the JSON fields of `EvaluationResult` or `StyleReport`. Malformed JSON results in a task retry or a `FAILED` status with an error log.
- **Orphan Task Sweeper**: A periodic Celery beat task queries the DB for `ExecutionTask`s stuck in `EXECUTING` for longer than a specified threshold (e.g., 1 hour), marking them as `FAILED (Timeout)` to prevent zombie states.

## 5. Performance Tuning

- **Database Connections**: Celery workers use connection pooling (via `django-db-geventpool` or similar) to prevent exhausting PostgreSQL connections during high-concurrency evaluation spikes.
- **Payload Size**: Large test outputs or massive code files are truncated before saving to `result_log` to avoid database bloat and memory pressure.
